# Outputs and lifecycle

Reference for `/impact-log` covering what the skill produces, what each entry looks like, where the log persists, and when to run each subcommand across an impact cycle. See the [main SKILL.md](../SKILL.md) for the Quick start and subcommand list.

---

## What you produce

The impact log is a **living document** for one individual contributor (IC) or engineering manager (EM) over an impact cycle (engineer, designer, data scientist, product manager, engineering manager, or any role that uses Spotify's Engineering Expectations or an equivalent framework). It accumulates structured evidence — one entry per meaningful work moment, including:

- Shipped deliveries
- Learnings that reshaped a plan or saved the team from wasted effort
- Postmortem insights that informed future work
- Near-misses with consequences worth capturing
- Scoping or design decisions that changed the trajectory of a project
- Cross-functional collaborations that yielded shared understanding
- **Valuable failures** — experiments that disproved a hypothesis, abandoned approaches that surfaced architectural insight, calls that turned out wrong but produced clarity. IC frameworks reward growth and learning; failures that produce value are evidence of both. Capturing them honestly (without spinning) is itself a signal of the kind of seniority that frameworks reward.

The `Outcome` field captures *what changed* — that includes new knowledge, not just new code or new artifacts. A learning that reframes a strategy is as much an outcome as a feature that ships.

The log renders into one of four destinations chosen at `init` time. It grows continuously and gets synthesized at the end of the cycle via `/team-update quarterly`.

| Stage | Artifact |
|---|---|
| In flight | `.impact-log/` directory in the IC's project workspace — local working state |
| Persisted | One destination chosen at `init`: GitHub private repo, Google Doc, Google Sheet, or local directory |
| Synthesized | Quarterly impact draft generated by `/team-update quarterly` against the log |

## The entry shape

Every entry has the same nine fields (defined in [`template-impact-log.md`](template-impact-log.md) and [`template-impact-log.csv`](template-impact-log.csv)):

| Field | What it captures | Example |
|---|---|---|
| Pillar | Business Impact / Technical Excellence / Culture & Organization | `Technical Excellence` |
| Sub-area | One of 9 (3 per pillar) | `Reliability` |
| Date | When the work shipped (YYYY-MM-DD) | `2026-04-29` |
| Title | Short label | `Cut p99 latency on /search by 40%` |
| Outcome | WHAT changed for users / business / team — written as an outcome, not an activity | `Search response time at p99 dropped from 850ms to 510ms; reduced timeout-driven retries by 60%` |
| AI accountability | If AI assistance was used, how was the output validated? | `Claude drafted the caching layer; I ran load tests against staging before merge.` |
| Blast radius / speed calibration | Reversible quick iteration vs. high-stakes rigor | `High-stakes — production search path; full review and staged rollout` |
| Source | Link to PR / doc / experiment / Slack thread | `https://ghe.spotify.net/.../pull/123` |
| Strategic alignment (optional) | Link to Tech Strategy / Company bet / OKR / SteerCo doc | `Tech Strategy 2026 — Reliability` |

## Sample entries across roles and outcome types

The field structure works the same way regardless of role or outcome type. The examples below show three different shapes of entry — one shipping delivery, one learning that reshaped a plan, one valuable failure where the proposal didn't ship but produced lasting value.

**Shipping delivery** (engineering, Technical Excellence / Reliability):

```markdown
## 2026-04-29 — Cut p99 latency on /search by 40%

**Pillar:** Technical Excellence
**Sub-area:** Reliability
**Outcome:** Search response time at p99 dropped from 850ms to 510ms; reduced timeout-driven retries by 60%.
**AI accountability:** Claude drafted the caching layer; ran load tests against staging before merge.
**Blast radius / speed calibration:** High-stakes — production search path; full review and staged rollout.
**Source:** https://ghe.spotify.net/.../pull/123
**Strategic alignment:** Tech Strategy 2026 — Reliability
```

**Learning that reshaped a plan** (data science, Business Impact / Decision quality):

```markdown
## 2026-03-15 — Showed dwell-time signal degrades cold-start ranking

**Pillar:** Business Impact
**Sub-area:** Decision quality
**Outcome:** Six-week analysis showed the dwell-time signal we'd planned to integrate adds noise rather than signal for users in their first 30 days. Findings led to a 30-day cold-start exemption now in scope for next cycle; saved the team from shipping a change that would have degraded cold-start engagement by ~4%.
**AI accountability:** Used Claude to spec the SQL queries and validate the analysis approach; ran two independent statistical tests against fresh holdout data before reporting.
**Blast radius / speed calibration:** Reversible — exploratory analysis; informs a future feature decision, no production change.
**Source:** Notebook (link), Slack thread with team (link)
**Strategic alignment:** Personalization Quality OKR — Cold-start engagement
```

**Valuable failure** (any IC role, Culture & Organization / Influence):

```markdown
## 2026-02-04 — Evaluation-framework proposal not adopted; design reviews produced lasting value

**Pillar:** Culture & Organization
**Sub-area:** Influence
**Outcome:** Proposed an evaluation framework for measuring an outcome the team had been measuring inconsistently. After two design reviews, the team chose not to adopt — the framework didn't decompose cleanly into per-decision actionability. The reviews surfaced a clearer mental model for what the team *does* want to measure (now informing a follow-up proposal), and the framing approach was reused by another contributor for a different evaluation problem.
**AI accountability:** N/A — proposal / decision flow, no implementation.
**Blast radius / speed calibration:** Low — proposal stage; reversible.
**Source:** Design doc (link), meeting notes (link)
**Strategic alignment:** —
```

## Format: Markdown vs CSV

- **Markdown** — used by `--to local` and `--to github`. Append new entries at the end. Human-readable; renders in any markdown viewer; diffable in git.
- **CSV** — used by `--to gsheet` to bootstrap a fresh Google Sheet via `gworkspace-write.sh --sheets-create-from-csv`. Same nine columns; one row per entry.

Both share the same nine columns. The choice between them is purely a function of where the log persists, not of what gets captured.

## Persistence destinations

| Destination | What renders | When to choose it |
|---|---|---|
| `--to github` | A markdown file in a personal private GitHub repo | If you already keep personal notes in git and want full history + offline access |
| `--to gdoc` | A Google Doc owned by you | If you want easy sharing with a manager during reviews and rich-text editing |
| `--to gsheet` | A Google Sheet owned by you | If you want filtering / sorting / pivot-table analysis across entries |
| `--to local` | A markdown file in a local directory | If you want maximum control and no cloud dependency |

The destination is decided at `init` and locked for that log. Switching later means manually re-running `persist` against the new destination.

## When to run each skill in an impact cycle

```
                                      Quarter
                                        ↓
 ┌───────────────────────────────────────────────────────────────────┐
 │ Setup                                                              │
 │   /impact-log init  ← once per cycle (or once total if persistent) │
 ├───────────────────────────────────────────────────────────────────┤
 │ Continuous capture                                                 │
 │   /impact-log add   ← each meaningful moment, as it happens        │
 ├───────────────────────────────────────────────────────────────────┤
 │ Periodic top-ups (weekly or monthly)                               │
 │   /impact-log add --from quarterly   ← bulk import from synthesis  │
 │   /impact-log add --from ghe         ← pull merged PRs from period │
 │   /impact-log add --from jira        ← pull tickets from period    │
 │   /impact-log rescue                 ← catch evidence about to     │
 │                                       expire (Slack 90d, etc.)    │
 │   /impact-log status                 ← see what's logged + gaps    │
 ├───────────────────────────────────────────────────────────────────┤
 │ Persist (any time, repeatable)                                     │
 │   /impact-log persist                ← push to chosen destination  │
 ├───────────────────────────────────────────────────────────────────┤
 │ End of cycle                                                       │
 │   /team-update quarterly             ← synthesizes log into draft  │
 │                                       impact doc                   │
 └───────────────────────────────────────────────────────────────────┘
```

**Rough cadence guidance:**

- **Daily**: `add` for any moment worth capturing
- **Weekly**: `status` to confirm coverage; `add --from quarterly` if you ran `/team-update quarterly` recently
- **Monthly**: `rescue` to catch ephemeral evidence; `add --from ghe` and `add --from jira` to backfill from PRs and tickets
- **Cycle end**: `persist` if you haven't recently; run `/team-update quarterly` for the synthesis
- **I&DT prep window** (May–June for July cycle, Nov–Dec for January cycle): `prep` to draft Workday-ready self-reflection; `prep --calibrate` for manager-lens self-check

The `/impact-log schedule install` subcommand can register a recurring `rescue` (or any other subcommand) via cron / launchd so the periodic top-ups run themselves.

**July vs January cycles:** The "End of cycle" stage looks different depending on the cycle. The July cycle is self-reflection + conversation only (no Impact Snapshot, no compensation). The January cycle includes Impact Snapshot (manager assessment on the Impact Scale) and compensation review. See [`idt-framework-employee.md`](idt-framework-employee.md) § Cycle Awareness for dates and details.

## Downstream uses

The log is the input to `/team-update quarterly`, which synthesizes it into a draft impact statement at the end of each impact cycle. Beyond that, the persisted destination is a structured document the IC owns and uses as they see fit.

The log persists across impact cycles unless the IC starts a fresh one each cycle.
